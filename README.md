# Task Manager

## Project Description

Task Manager is a small task management app built with plain HTML, CSS, vanilla JavaScript, and a Node.js + Express backend.

The backend in `server.js` serves the frontend UI from the `public` folder and exposes task API routes under `/api/tasks`.

Task data currently uses temporary in-memory backend storage. This means tasks are stored only while the Node server is running. The theme preference still uses browser `localStorage` because it is only a local UI setting.

## Features

- Add tasks from a modal form.
- Edit existing tasks.
- Delete tasks.
- Confirm before deleting a task.
- Mark tasks complete or incomplete.
- Clear all completed tasks.
- Search tasks by title.
- Focus search with the `Ctrl+K` shortcut. On macOS, `Cmd+K` is also handled by the current JavaScript.
- Filter tasks by All, Active, and Completed.
- Use sidebar views for My Day, Important, Planned, Completed, and All Tasks.
- Collapse and expand the sidebar.
- Create or edit tasks through a 3-step task modal: title, dates, and priority.
- Add optional start date and end date values.
- Validate that a task title is required.
- Validate task title length with an 80-character limit.
- Validate that the end date is not earlier than the start date.
- Show custom validation and error messages.
- Choose task priority: Low, Medium, or High.
- Toggle between light and dark themes.
- Show toast messages for actions and errors.
- Show tasks due today in My Day.
- Show dated tasks in Planned.
- Treat high-priority tasks as Important.
- Mark incomplete tasks as overdue when their end date is before today.
- Use a responsive layout with automatic sidebar behavior on smaller screens.

## Install and Run

Install dependencies and start the watch-mode development server:

```bash
npm install
npm run dev
```

Then open:

```text
http://localhost:3000
```

For a non-watch server, run:

```bash
npm start
```

If PowerShell blocks the npm shim, use the `.cmd` fallback:

```bash
npm.cmd install
npm.cmd run dev
```

## Backend Routes

All task routes are served under `/api/tasks`.

```text
GET    /api/tasks       Get all tasks
POST   /api/tasks       Create a task
PUT    /api/tasks/:id   Update a task
DELETE /api/tasks/:id   Delete a task
```

### POST `/api/tasks`

Creates a new task.

Example request body:

```json
{
  "title": "Plan the week",
  "startDate": "2026-04-29",
  "endDate": "2026-04-30",
  "priority": "medium"
}
```

### PUT `/api/tasks/:id`

Updates an existing task.

Example request body:

```json
{
  "title": "Plan the week",
  "completed": true,
  "startDate": "2026-04-29",
  "endDate": "2026-04-30",
  "priority": "high"
}
```

### Error Format

Errors are returned as JSON:

```json
{
  "error": "Task title is required."
}
```

### Status Codes

- `200` for successful `GET`, `PUT`, and `DELETE` requests.
- `201` for a successful `POST` request.
- `400` for invalid input.
- `404` when a task cannot be found.
- `500` for a server error.

## Task Data Model

Task data includes:

```js
{
  id: "unique-id",
  title: "Task title",
  completed: false,
  startDate: "",
  endDate: "",
  dueDate: "",
  priority: "low | medium | high",
  createdAt: "ISO timestamp",
  updatedAt: "ISO timestamp or null"
}
```

Notes:

- `id` is generated by the backend.
- `createdAt` is generated when a task is created.
- `updatedAt` starts as `null` and changes when a task is edited.
- `startDate` and `endDate` are used by the current UI.
- `dueDate` is kept for compatibility and currently mirrors `endDate` in the backend.

## Storage Limitations

The backend currently uses an in-memory array in `server.js`.

Tasks are available while the server is running, but they disappear when the server restarts. This is intentional for the first backend step so the app can stay simple while the API and frontend behavior are being built.

A database such as MySQL can be added later for persistent task storage.

The theme preference still uses browser `localStorage` because it is only a local UI setting.
